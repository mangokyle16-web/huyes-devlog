# 驚喜提案六:拼配副駕 Blend Copilot —— 用光譜「複製」一支賣到斷貨的招牌配方

---

## 你以為配方是食譜,其實它是「會消失的食譜」

烘豆廠最賺錢、最有品牌價值的東西不是單品豆,而是**招牌配方(house blend)**。但配方有個致命弱點:

> **配方裡的某支豆用完了 / 那一季的批次風味漂了 / 產地缺貨了。**

這時烘豆師只能憑舌頭和經驗,在一堆現有庫存裡「試拼」——調比例、烘、杯測、再調,一輪要好幾天,而且換人就換味道。**配方不是穩定的食譜,而是每次補貨都要重做一次的猜謎。**

這是一個**反覆發生、有人付錢解決、目前只能靠老師傅手感**的硬需求。

---

## 核心構想:把「拼配」變成一個光譜最佳化問題

你已經在累積每一批豆的 10-band 光譜指紋(密度模型、瑕疵閘門都在採)。Blend Copilot 把拼配反過來做:

> **給定一個「目標光譜指紋」(可以是上一批賣得很好的配方、或一支斷貨豆),系統從你「現有庫存」中,自動算出「哪幾支豆、各佔多少比例」最接近那個目標。**

三種使用情境:
1. **複製模式**:招牌配方的某支豆斷貨 → 用現貨拼出最接近原配方光譜的替代組合。
2. **匹配模式**:想復刻一支喝過、但買不到的豆 → 拼出最接近它的組合。
3. **設計模式**:給一個參考目標(色澤/烘焙度/某產地特徵)→ 反推配方。

---

## 技術架構

### 一句話
拼配 = 各批豆光譜向量的**加權組合**;求一組權重(比例)讓組合後的光譜最接近目標 → 這是個有約束的最小二乘 / 凸優化問題。

```
庫存每批豆 → 10-band 聚合光譜向量 v_i (已在採集 pipeline 產出)
目標 → 目標光譜向量 v_target
求 weights w_i (w_i ≥ 0, Σw_i = 1)
最小化 ‖ Σ w_i·v_i − v_target ‖
→ 輸出建議配方:豆 A 40% / 豆 B 35% / 豆 C 25%,與「光譜貼合度」分數
```

### 三層
```
Layer 1: 庫存光譜盤點(Pi5)
  ├─ 每進一批豆,輸送帶掃描,存 batch 10-band 聚合向量 + Agtron + 密度
  └─ 沿用現有 spectral_capture ledger,新增 inventory 標記

Layer 2: 拼配最佳化(Mac/雲端)
  ├─ scipy/cvxpy 解約束優化(權重非負、總和 1、可加最小/最大比例限制)
  ├─ 支援「鎖定某支豆比例」「排除某支」等實務約束
  └─ 輸出 Top-3 候選配方 + 各自貼合度

Layer 3: 輸出與閉環
  ├─ 手機 cockpit / 看板顯示建議配方
  ├─ 烘豆師實拼一小批 → 掃描 → 比對實際 vs 預測光譜
  └─ 杯測回饋修正(光譜貼合 ≠ 風味貼合,需校正)
```

---

## 分階段計劃(含 Gate)

### Phase 0:驗證「光譜可加性」(一個下午,Gate:混合光譜 ≈ 成分加權平均)
**這是整個提案的生死 gate。** 拼配優化的前提是:50/50 實體混豆的光譜,約等於兩支豆光譜的平均。
- [ ] 取兩支光譜差異明顯的豆,各掃單獨光譜。
- [ ] 實體 50/50 混合掃描,比對「實測混合光譜」vs「兩者平均」。
- [ ] **Gate GO:誤差在可接受範圍(各 band <一定 %)→ 可加性成立,繼續。**
- [ ] **Gate NO-GO:嚴重非線性 → 改用查表/回歸建混合模型,或暫停。**
> 代價:一個下午、零硬體成本。今天就能測。

### Phase 1:庫存光譜資料集(1–2 週)
- [ ] 把現有/新進每批豆掃描存成 inventory 向量(沿用採集 pipeline)。
- [ ] 至少 8–12 批不同豆,涵蓋常用配方成分。
- [ ] **Gate:同一批豆隔日重掃,向量穩定(批內可重現)。**

### Phase 2:最佳化器 + UI(3–5 天)
- [ ] `blend_optimizer.py`:約束最小二乘 + 比例約束 + Top-N 候選。
- [ ] cockpit「拼配」分頁:選目標 → 顯示建議配方 + 貼合度。
- [ ] **Gate:對一個已知配方,系統能從打散的成分庫存還原出接近原比例。**

### Phase 3:杯測閉環驗證(1–2 週)
- [ ] 用系統建議配方實拼數組,杯測比較 vs 原配方。
- [ ] **Gate:光譜貼合度高的配方,杯測相似度顯著高於隨機拼配。**

---

## 三個誠實 Caveats

1. **光譜貼合 ≠ 風味貼合。** VNIR 350–950nm 主要捕捉色澤/烘焙度/產地光學特性,不是芳香揮發物。Blend Copilot 給的是「光譜上最接近」的起點,**大幅縮短試拼次數**,但最後一哩仍需杯測收斂——定位成「副駕」不是「自動駕駛」。
2. **可加性必須先驗證(Phase 0)。** 若混合光譜嚴重非線性,線性優化就不成立,要改模型。
3. **垃圾進垃圾出。** 庫存向量的採集條件(光照/曝光/校正)要一致,否則優化在拼噪音。

---

## 為什麼現在做

1. **你已經在累積庫存光譜了** —— 密度模型、Green Bean Gate 的採集都在產出 10-band 批次向量,Blend Copilot 幾乎是**免費搭在同一份資料上的第二個產品**。
2. **這是 B2B 付費點** —— 烘豆廠為「配方穩定性」付錢的意願,遠高於消費者挑豆。一個能在斷貨時 30 分鐘給出替代配方的工具,直接省下老師傅好幾天的試拼。
3. **跟既有提案不重疊** —— 計數/重量(提案15)、瑕疵閘門(Green Bean Gate)、新鮮度(提案5)都是「單批診斷」;Blend Copilot 是第一個做「**跨批組合最佳化**」的方向,把你累積的光譜庫從「檔案」變成「能生成配方的引擎」。
4. **Phase 0 今天就能驗**,零成本,訊號不在就止損。
