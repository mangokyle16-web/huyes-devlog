# 驚喜提案三：Live Roast Copilot — 從「事後檢測」到「烘焙中即時引導」

> 撰寫時間：2026-06-03 深夜（自主思考）
> 前兩份：BQS Q-Grader（提案一）、SBIR × 嘖嘖（提案二）
> 本份：完全不同的技術方向

---

## 核心洞察：你一直在測量「已經發生的事」

現在的 Huyes 流程：

```
生豆 → 烘焙（黑箱）→ 熟豆 → Huyes 測量 Agtron → 太深了？太淺了？→ 只能接受
```

**問題：烘焙過一次就回不去了。**

Huyes 現在是「驗屍報告」，不是「生命監測」。

---

## 提案：把相機對準烘焙機滾筒

```
生豆 → 烘焙中 → Huyes 相機即時監測 → 預測還有幾秒到目標 Agtron → 警報下鍋
```

**同一台相機，換個安裝位置，變成完全不同的產品。**

---

## 為什麼技術上可行（現有資源直接用）

### 現有 Huyes 技術的對應關係

| 現有技術 | Live Roast 用途 |
|---------|----------------|
| CM020D 多光譜相機 | 拍攝滾筒出口落豆區（耐熱距離 30cm+ 即可）|
| 10-band 光譜分析 | 豆子顏色頻譜隨烘焙度連續變化 |
| Agtron 校正公式 | 已知光譜→Agtron 的映射關係 |
| Pi5 即時運算 | 每秒拍攝 + 分析，延遲 < 500ms |
| Huyes App | 顯示即時曲線 + 倒數計時 |

### 光譜變化的物理基礎

咖啡豆在烘焙過程中有明確的光譜變化規律：

```
生豆（綠）→ 一爆前（黃）→ 一爆（焦糖化）→ 二爆前（深褐）→ 二爆（油脂滲出）
  ↓              ↓                ↓                  ↓               ↓
高反射率       850nm 下降      930nm 相對升       全波段暗化       油脂造成反射峰
```

Huyes 的 FPI 相機正好覆蓋 850nm 和 930nm——**這不是巧合，這是天然的終點偵測器。**

## 產品形態：Huyes Roast Copilot

### 硬體（極簡改動）

```
現有 Huyes 套組
  + 耐熱鏡頭保護套（NT$500 零件）
  + 磁吸式滾筒出口夾具（3D 列印，已有能力）
  + 散熱風扇（Pi5 GPIO 控制，已會）
```

**額外硬體成本 < NT$2,000**

### 軟體新增（Mac Mini 可直接開發）

1. **`roast_monitor.py`（Pi5 執行）**
   - 每 0.5 秒拍攝一幀
   - 提取落豆區 ROI 的平均光譜
   - 計算即時 Agtron 估算值
   - 偵測一爆聲（麥克風 + GPIO，Pi5 已有能力）

2. **`roast_predictor.py`（Mac Mini 模型）**
   - 輸入：目前光譜曲線斜率
   - 輸出：預測幾秒後到達目標 Agtron
   - 模型：線性回歸（資料量小，不需要深度學習）

3. **Huyes App 新增「烘焙模式」Tab**
   - 即時 Agtron 曲線圖
   - 目標線（使用者設定）
   - 倒數計時 + 警報

## 為什麼這個方向讓競爭者很難複製

### 現有競品

| 產品 | 原理 | 價格 | 缺點 |
|------|------|------|------|
| Cropster / Artisan | 只記錄溫度曲線 | NT$15,000+/年 | 無光譜，靠經驗判斷 |
| ColorTrack（Probat） | RGB 顏色感測 | 僅大型機台 | 只有 3 個顏色通道 |
| 人工目視 | 眼睛 | - | 主觀、疲勞 |

**Huyes 優勢：10 個光譜通道 vs 競品的 3 個 RGB**

RGB 相機在深色豆子上飽和，看不出差異。多光譜在 NIR 波段仍然有清晰的區分力。

## 商業模式轉變

### 從 C 端轉向專業 B 端

| | OPTIC-BEAN SORTER（原定）| Roast Copilot（新） |
|--|--------------------------|---------------------|
| 客群 | 消費者、咖啡愛好者 | 精品咖啡烘焙師 |
| 決策週期 | 衝動購買（嘖嘖）| 試用後採購 |
| 單價 | NT$10,000-30,000 | NT$50,000-120,000 |
| 量 | 需要大量 | 台灣 500+ 精品烘焙廠 |
| 黏性 | 低 | 極高（每天使用）|
| 與 SBIR 的結合 | 消費品 | **食品安全 + AI 技術** |

**Roast Copilot 的 SBIR 申請理由更強：**
「即時阻止過度烘焙造成的黃麴毒素活化風險」——食安角度，經濟部最愛。

## 與現有專案的協同

這個方向不是重來，而是**擴展現有工作**：

```
現有 Agtron 校正 → Roast Copilot 的核心模型
現有 FastSAM 分割 → 改為即時 ROI 提取
現有 Pi5 pipeline → 改為即時串流模式
現有 Huyes App → 新增 Roast Mode Tab
現有 BQS 系統 → 烘焙結束後自動跑 BQS，形成完整閉環
```
