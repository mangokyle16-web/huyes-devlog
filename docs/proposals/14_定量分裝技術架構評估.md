# 提案 14：Plan D 定量分裝技術架構評估

## 結論

Plan D 可行，但它不是「99% counting accuracy 直接等於 99% weight accuracy」。現有系統已經有很好的即時計數基礎：YOLOX-tiny -> `BeanTracker` -> `/dev/shm/count_status.json` -> phone cockpit。要把它變成 dosing / portioning machine，真正的架構挑戰在三件事：

1. `count -> grams` 的校正模型。
2. counting line 到 actuator 之間的 in-flight beans overshoot。
3. GPIO actuator 的可靠機構與 fail-safe control loop。

推薦架構：

- **v1**：count-based dosing，使用每批 `g_per_bean` 固定校正，`dosing_controller.py` daemon 讀 `/dev/shm/count_status.json`、執行 feed-forward early trigger、用 GPIO 控制 conveyor motor stop 或簡單 gate。預期實測重量精度約 **+/-5-10%**，調好機構與校正後可望到 **+/-3-6%**。
- **v2**：加入 load cell + HX711 做 closed-loop correction，並評估 YOLOX bbox area 的 size-aware weight estimate。目標精度才有機會穩定進入 **+/-1-2%**。

## 既有系統邊界

已完成並可沿用：

- Raspberry Pi 5 + Hailo-8 NPU + multispectral camera。
- `spectral_capture/preview_display.py` 是 live counting owner。
- `BeanTracker` 已做 global-motion compensation 與 dual counting line，`total_crossed` 約 99% accurate，實測例：403 vs 399。
- `/dev/shm/count_status.json` 已每 0.5s 寫入 counting snapshot，包含 `total_crossed`, `new_crossings`, `live_tracks`, `fps`, `belt_speed_px_s`, `updated_at`。
- `spectral_capture/control_server.py` 已有 FastAPI，port `8765`，phone cockpit 透過 Tailscale 使用。
- `spectral_capture/ui/index.html` 已有「計數」tab 與 `/api/count`, `/api/batches` polling。

Plan D 應該把這些當成穩定輸入，不要把 dosing logic 塞回 detector 或 phone UI。

## 1. Weight Estimation Model

### 問題本質

咖啡豆平均單顆重量變異很大，約 `0.10-0.22 g/bean`，取決於品種、處理法、烘焙度、水分、大小分布、破碎豆比例。Counting accuracy 只回答「幾顆」，不回答「每顆多重」。

基本公式：

```text
estimated_weight_g = total_crossed * g_per_bean
```

若 `total_crossed` 有 99% accuracy，但 `g_per_bean` 估錯 8%，最後重量仍會錯約 8%。因此 Plan D 的 v1 accuracy ceiling 主要由 calibration 與 actuator overshoot 決定，不是由 YOLOX 決定。

### Option A：每批固定 calibration constant

流程是每一批豆先手動數或系統數 `N_cal` 顆，例如 100 或 200 顆，放到電子秤得到 `W_cal`：

```text
g_per_bean = W_cal / N_cal
target_count = target_g / g_per_bean
```

優點：

- 最簡單，與現有 `total_crossed` 最匹配。
- 不需要改 YOLOX model，也不需要把 bbox history 做成 durable event stream。
- 若 calibration sample 代表該批豆，對 250g 這種大份量會比直覺上準，因為單顆隨機大小差異會在大量 beans 上平均掉。

限制：

- 如果只用跨批固定值，例如永遠假設 `0.16 g/bean`，不同豆種或烘焙度可能造成 **+/-10-30%** error。
- 如果每批都用 100 beans 校正，理想情況下模型 error 可能降到 **+/-2-5%**，但實機仍會被 sampling bias、破碎豆、clumping、actuator overshoot 拉回 **+/-5-10%**。
- 小份量比大份量更難準。例如 18g espresso dose 只有約 80-180 顆，單顆 variance 與 overshoot 影響更明顯；250g 則較容易平均。

### Option B：用 YOLOX bbox area 做 size-aware estimate

概念是把每顆 crossing event 的 bbox 面積當作體積 proxy：

```text
bean_weight_g = alpha * bbox_area + beta
estimated_weight_g = sum(bean_weight_g for crossed beans)
```

或使用更穩定的特徵：

```text
feature = sqrt(width * height), width, height, area, aspect_ratio
```

優點：

- 理論上可以補償「大豆比較重、小豆比較輕」。
- 若 camera geometry、belt height、lighting 固定，bbox size 對同一套機構有可用訊號。
- 對混合大小分布、破碎豆比例高的 batch，可能比固定平均值好。

限制：

- bbox area 是 2D 投影，不是 volume。豆子姿態、重疊、陰影、相機高度、focus、threshold 都會影響面積。
- 現有 `count_status.json` 只暴露 aggregate count，不暴露 per-crossing bbox feature。要做 size-aware，需要 `preview_display.py` 或 `BeanTracker` 在 crossing event 時產生 per-bean event。
- 沒有 scale ground truth 時，bbox area model 會很容易看起來科學但不準。
- 需要每種 camera setup 重新校正，移動鏡頭或改 belt width 都會漂移。

預期精度：

- 未校正或只做簡單線性模型：不一定比 fixed per-batch mean 好。
- 有 scale ground truth、固定機構、每批或每類豆校正後：可能把 v1 的 **+/-5-10%** 改善到 **+/-3-6%**，但很難保證 **+/-1-2%**。

### Option C：Hybrid

推薦長期方向：

```text
estimated_weight_g =
  total_crossed * batch_g_per_bean
  + size_adjustment_from_bbox_area
```

Hybrid 讓 per-batch `g_per_bean` 決定主要尺度，bbox area 只做相對修正。這比純 bbox model 穩，因為它不要求 camera 直接預測絕對重量。

### 推薦 calibration workflow

v1 應採用 Option A：

1. 每批開始前拿 100 或 200 顆校正樣本。
2. 用電子秤稱總重，輸入 `calibration_count` 與 `calibration_weight_g`。
3. 系統計算 `g_per_bean`，保存到 dosing setpoint。
4. dosing 時使用 `target_count = floor(target_g / g_per_bean)`，再套用 in-flight early trigger correction。
5. 每次 dose 結束後，若使用者有實際稱重，輸入 `actual_weight_g`，記錄 calibration error，用於後續調整 `g_per_bean`。

v2 才加入 load cell。若產品目標是賣「定量分裝」而不是「大概分裝」，load cell 最終應該在 loop 裡。Counting 是快速 feed-forward，load cell 是慢速 truth source。

## 2. Control Loop 與 Latency

### 為什麼會 overshoot

現有 belt left-to-right，速度約 `160 px/frame`。Detection rate 約 `5-6 fps`，所以 belt speed 約：

```text
160 px/frame * 5.6 frame/s = 896 px/s
```

如果 counting line 到 gate / bagging point 距離是 `D_px`，bean 從 counting line 到 actuator point 的 transit time：

```text
transit_time_s = D_px / belt_speed_px_s
```

例如 `D_px = 600 px`，`belt_speed_px_s = 896 px/s`：

```text
transit_time_s = 0.67s
```

這 0.67 秒內已經越過 counting line 但還沒到 gate 的 beans，就是 in-flight beans。當 software 發現 target reached 再觸發 gate，實際落袋會多出這段距離上的 beans。

### Timing budget

粗略 budget：

| Component | v1 估計 |
|---|---:|
| Camera frame interval | 167-200 ms at 5-6 fps |
| YOLOX + tracker processing | 約 10-50 ms，視 pipeline scheduling |
| `count_status.json` publish interval | 最多 500 ms，現有 contract |
| `dosing_controller.py` polling interval | 建議 20-50 ms |
| GPIO output latency | < 5 ms |
| Relay / motor driver electrical response | 5-30 ms |
| Servo gate mechanical travel | 150-400 ms |
| Solenoid gate mechanical response | 20-80 ms |
| Conveyor inertia after motor stop | 100-1000 ms，取決於馬達、皮帶、負載 |

最大問題不是 GPIO，而是 frame cadence、現有 0.5s aggregate publish interval，以及 actuator mechanics。

### 推薦 feed-forward correction

不要等 `estimated_weight_g >= target_g` 才觸發。應該預測：

```text
beans_in_flight =
  throughput_beans_per_s * (transit_time_s + actuator_delay_s)

trigger_count =
  target_count - round(beans_in_flight) - safety_margin_beans
```

若使用重量目標：

```text
target_count = target_g / g_per_bean
trigger_weight_g = trigger_count * g_per_bean
```

其中：

- `transit_time_s` 由 `gate_x_px - counting_line_x_px` 與 `belt_speed_px_s` 推算。
- `actuator_delay_s` 由實測設定，例如 servo gate 0.25s、solenoid 0.05s、motor stop 0.4s。
- `throughput_beans_per_s` 可以用最近 1-3 秒 crossing rate，比 60 秒 throughput 更適合 control。
- `safety_margin_beans` v1 可先設 0-3 顆，透過實測調整。

這是 feed-forward control，不是 closed-loop。它可以顯著降低 overshoot，但不能消除機構變異。若要精準到 +/-1-2%，仍需要 load cell 或終端感測確認。

### `count_status.json` publish rate 需要注意

現有 0.5s publish rate 對 phone cockpit 足夠，但對 dosing trigger 偏慢。架構上有兩個選項：

- v1a：`dosing_controller.py` 先讀現有 `/dev/shm/count_status.json`，接受較大 overshoot，優先驗證硬體。
- v1b：讓 counting owner 另寫 `/dev/shm/count_events.jsonl` 或提高 dosing-only status update rate 到 20-50Hz。

推薦 v1 先用現有 status 完成 bench proof，但正式 dosing v1 應增加 event-level 或 faster status path。Phone UI 仍維持 1s polling，不需要吃高頻資料。

## 3. Actuator Options

### Option A：Stop conveyor motor

用 GPIO 控制 relay、MOSFET、或 motor driver enable pin，達標時停止 belt。

優點：

- 機構最簡單，solo-maker 最容易先做出來。
- 不需要設計高速 gate timing。
- 適合「一次倒一批，達標停機，人工換袋/取杯」的 bench dosing。

缺點：

- Conveyor inertia 造成 overshoot，尤其 DC motor + belt 沒有 braking 時。
- 停帶後 belt 上仍有 beans，下一次 dose 的初始狀態要清楚定義。
- 頻繁 start/stop 可能造成 clumping 或流量不穩。

GPIO / driver：

- 小 DC motor：不要直接接 Pi GPIO，使用 motor driver 或 MOSFET driver。
- Relay module 可用於簡單 on/off，但不適合高頻切換。
- 若 motor driver 支援 `enable`, `brake`, `PWM`，比 relay 好。
- 必須有 independent e-stop，切斷 motor power，不依賴 Python process。

### Option B：Servo / solenoid gate

Gate 平常關閉或導向回收，dosing 時打開流入 bag，到 target 後關閉或切到 reject/return path。

優點：

- Conveyor 可以保持穩定速度，counting 與 flow 更穩。
- 比 stop motor 更適合連續分裝，多袋連續裝填。
- Gate 可以放在 counting line 下游，配合 transit-time correction。

缺點：

- 需要機構設計，gate 的死角、卡豆、反彈會影響可靠性。
- Servo 速度慢但控制簡單；solenoid 快但需要電源、driver、flyback protection。
- Gate 關閉瞬間可能夾豆，造成破碎或 jam。

GPIO / driver：

- Servo：使用 PWM，建議獨立 5V power，Pi GPIO 只給 signal，共地。
- Solenoid：MOSFET + external power + flyback diode，不能直接接 GPIO。
- Gate position 最好 v2 加 limit switch 或 Hall sensor，避免「軟體以為關了但實際卡住」。

### Option C：Pneumatic diverter

用 air jet 或氣缸把 beans 分流。

優點：

- 反應快，適合高速分選與多通道 sorting。
- 與未來瑕疵豆剔除架構相容。

缺點：

- 需要 compressor / pump / valve，噪音、成本、可靠度都上升。
- Bench consumer dosing 過度複雜。
- 對單純 250g portioning，不是 v1 最小可行路徑。

GPIO / driver：

- Solenoid valve driver、external power、flyback diode 或 TVS。
- 需要 air pressure regulation 與安全洩壓。

### Option D：Vibratory feeder gate

用 vibratory feeder 控制落豆速度，接近 target 時降速或關閉。

優點：

- 最適合 precision dosing。接近 target 時可以 slow feed，減少 overshoot。
- 與 load cell closed-loop 非常搭。

缺點：

- 機構與控制比 conveyor stop 複雜。
- 需要調振幅、頻率、料槽角度，初期 bring-up 時間較長。

GPIO / driver：

- 需要 motor driver / MOSFET / PWM 或專用 vibratory driver。
- 最好與 load cell 搭配，不然 precision 優勢發揮不出來。

### v1 actuator recommendation

如果目標是 solo-maker consumer / bench prototype，推薦：

1. **第一個實驗：motor driver enable/brake 停 conveyor**。這是最快驗證 count-to-dose 的方式。
2. **第一個較像產品的 v1：simple servo gate 或 solenoid gate**。Conveyor 穩定跑，gate 控制袋口進料，使用 feed-forward correction。

若只能選一個做 v1，我會選 **servo/solenoid gate**，但前提是有簡單可靠的滑道或漏斗機構。若機構還沒做，先用 motor stop 做 bench proof。

## 4. Software Architecture

### 選項比較

| Option | 評估 |
|---|---|
| Extend `preview_display.py` | 不推薦。它已經是 pygame UI、detector、tracker、batch save owner。再加 GPIO dosing 會讓 UI crash 影響 actuator，責任過重。 |
| Put dosing in `control_server.py` | 不推薦作為 control owner。FastAPI 適合設定 target 與顯示 status，不應該直接負責 real-time GPIO action。server restart 或 request latency 不該影響 actuator。 |
| Dedicated `dosing_controller.py` daemon | 推薦。它是小型 control process，讀 count status，擁有 GPIO，寫 dosing status，FastAPI 只當 API facade。 |

### 推薦架構

```text
preview_display.py
  YOLOX + BeanTracker
  writes /dev/shm/count_status.json
        |
        v
dosing_controller.py
  reads count_status
  reads dosing_setpoint
  owns GPIO actuator
  writes dosing_status
        |
        v
control_server.py
  POST /api/dosing/setpoint
  POST /api/dosing/start
  POST /api/dosing/stop
  GET  /api/dosing/status
        |
        v
phone cockpit
  target input + progress + actuator status
```

### 為什麼 `dosing_controller.py` 要獨立

- GPIO ownership 單一化，避免 `preview_display.py`、FastAPI、測試 script 同時拉同一個 pin。
- 可用 systemd 讓它開機常駐，crash 後重啟時先進 `SAFE_OFF`。
- 可用 tighter loop，例如 20-50ms polling，而 phone UI 不需要高頻。
- 可以把 safety rule 寫在一個地方：stale count、lost preview、e-stop、target invalid、actuator timeout。

### Control state machine

```text
IDLE
  -> ARMING
  -> DOSING
  -> TARGET_REACHED
  -> ACTUATE
  -> SETTLE
  -> DONE
  -> RESET -> IDLE

Any state -> ERROR
Any state -> ABORTED
```

State meaning：

| State | 說明 |
|---|---|
| `IDLE` | 無 active target，actuator 在 safe state。 |
| `ARMING` | 檢查 setpoint、calibration、count status freshness、GPIO ready、e-stop。 |
| `DOSING` | 持續讀 `total_crossed`，計算 `estimated_weight_g` 與 `trigger_count`。 |
| `TARGET_REACHED` | 軟體判定已到 early-trigger threshold，freeze trigger decision，避免重複觸發。 |
| `ACTUATE` | 發 GPIO command，例如 stop motor、close gate、divert gate。 |
| `SETTLE` | 等待 beans in flight 落完，收斂 final count。 |
| `DONE` | dose 完成，status frozen，等待 user reset 或下一袋。 |
| `ABORTED` | 使用者 stop 或 e-stop。actuator 進 safe state。 |
| `ERROR` | count stale、GPIO failure、actuator timeout、invalid calibration 等。actuator 進 safe state。 |

Phone cockpit should remain command-and-observe UI，不做 real-time control loop。

- Target mode segmented control：`grams` / `beans`。
- Input：
  - `target_g`
  - `target_count`

  - `Set target`

新增 FastAPI endpoints：

|---|---|---|


## 6. Calibration 與 Accuracy Realism

### Error budget

v1 的重量誤差來源：

| Source | Typical impact |
|---|---:|
| Counting error | 約 +/-1%，依當前 99% 實測 |
| Per-batch `g_per_bean` calibration | 約 +/-2-5%，視樣本代表性 |
| Bean size / moisture drift inside batch | 約 +/-1-5% |
| In-flight overshoot correction | 約 +/-1-8%，視 actuator 與 flow |
| Clumping / merged detections | 低流量時小，clumping 時可變大 |
| Actuator repeatability | motor stop 可能最大，solenoid/gate 次之 |

所以：

- **v1 no scale**：對 250g 這種中大份量，合理承諾是 **+/-5-10%**。調好 calibration、流量、feed-forward 後可追求 **+/-3-6%**。
- **v1 不應承諾 +/-1-2%**。這會被 calibration 與 mechanics 打破。
- **v2 with load cell**：若使用 slow final correction，例如接近 target 時降速或 pulse feed，再用 scale stop，可以追求 **+/-1-2%**。

### Load cell 是否必要

看產品承諾：

- 若用途是粗分裝、教學 demo、batch pre-portion，count-based v1 足夠。
- 若用途是「消費者買回家設定 250g，希望每袋都接近標稱重量」，需要 load cell。
- 若用途是 espresso dosing 或小份量高精度，load cell 幾乎必須要有。

推薦：v1 不放 load cell，先驗證 count-to-actuator 與 overshoot model；v2 加 load cell，不要在 v1 同時背負機構、GPIO、scale、closed-loop 的全部風險。

## 7. Risks 與 Open Questions

### Risks

- **Belt jams**：gate close、豆子堆積、滑道角度不足都會 jam。需要 timeout 與 manual clear mode。
- **Bean clumping**：多顆貼在一起可能被 YOLOX merge 成一個 bbox，count 和 bbox area 都會錯。
- **Actuator reliability**：servo stall、solenoid heat、relay contact wear、motor inertia 都會影響 repeatability。
- **Power loss mid-dose**：重啟後不知道哪些 beans 已落袋。v1 應標記 last dose `ERROR_RECOVERY_REQUIRED`，要求人工清空與 reset。
- **Stale count status**：`preview_display.py` crash 或 detector 卡住時，dosing must fail closed。
- **GPIO conflict**：所有 GPIO control 必須由 `dosing_controller.py` owns，避免 test script 或 FastAPI 同時控制。
- **Mechanical coordinate calibration**：`gate_x_px` 與 `counting_line_x_px` 不是文件常數，必須用實測標定。
- **Throughput variation**：beans flow 不是連續流體，短時間 rate 很 noisy。feed-forward 要用 recent crossing rate + margin，而不是只用 60 秒平均。

### Open questions

- Counting line 到實際 gate / bag drop point 的 pixel distance 是多少？
- Conveyor motor 是否支援 brake 或只支援 power cut？
- 目前 flow rate 是多少 beans/s？是否穩定，還是會一陣一陣落豆？
- Actuator 要做「停帶」還是「分流到袋口」？
- v1 target 是 250g 大份量，還是 18-30g 小份量？
- 是否接受 v1 需要手動秤 100/200 beans 做每批 calibration？
- 系統斷電後，機構 safe state 是 gate closed、motor off，還是 diverter to reject？

## v1 Architecture Recommendation

### Weight model

使用 per-batch fixed `g_per_bean`：

```text
g_per_bean = calibration_weight_g / calibration_count
target_count = target_g / g_per_bean
```

不要在 v1 做 bbox size-aware model。v1 可以開始記錄 bbox features，但不要讓它控制 dosing，避免一開始把錯誤來源變多。

### Actuator

首個 bench experiment 用 motor stop 驗證 end-to-end。若要做像產品的 v1，改成 servo/solenoid gate：

- Conveyor stable speed。
- Gate controls bag fill。
- `dosing_controller.py` early-trigger closes/diverts gate。

### Software

新增 dedicated `dosing_controller.py` daemon：

- Reads `/dev/shm/count_status.json`。
- Reads `/dev/shm/dosing_setpoint.json`。
- Owns GPIO。
- Writes `/dev/shm/dosing_status.json`。
- FastAPI 只提供 `/api/dosing/*` facade。
- Phone cockpit only sets targets and observes status。

### v1 BOM sketch

Motor-stop bench BOM：

- Raspberry Pi 5 GPIO breakout wires。
- Motor driver with enable/brake input，或 logic-level MOSFET module for DC motor。
- External motor power supply。
- Fuse。
- Physical e-stop switch cutting motor power。
- Optional opto-isolated relay module for low-frequency simple on/off。

Gate-based v1 BOM：

- SG90 / MG90S servo for slow gate，或 small solenoid for faster gate。
- Dedicated 5V servo power supply，或 solenoid-rated external power。
- MOSFET driver if solenoid。
- Flyback diode / TVS for inductive load。
- Common ground with Pi。
- 3D printed chute / funnel / gate。
- Physical e-stop。
- Optional limit switch for gate closed/open feedback。

## v2 Architecture Recommendation

v2 加兩個東西：

1. **Load cell closed-loop**：HX711 + load cell under bag/cup。Counting gives fast prediction，scale gives final truth。接近 target 時 conveyor slow down 或 gate pulse feed。
2. **Size-aware estimate**：在 crossing event 記錄 bbox area / width / height / score / track id，離線用 actual scale data 校正。若模型在 holdout batches 上明顯優於 fixed mean，再進 control loop。

v2 control pattern：

```text
bulk feed by count prediction
  -> near target early
  -> slow feed / pulse gate
  -> load cell final stop
  -> settle and record actual_weight_g
```

這才是能穩定賣「準確克重」的架構。

## Single Most Important Next Experiment

先不要做完整 UI 或 size-aware model。下一個最重要實驗是：

> 在實機上量測 `counting_line -> actuator point` transit time、actuator delay、以及 motor stop/gate close 後的 overshoot beans 分布。

最小實驗：

- 固定一個 target count，例如 100 beans。
- 跑 20 次。
- 記錄 `trigger_count`, final `total_crossed`, 實際落袋 weight。
- 分別測 motor stop 與簡單 gate。
- 用結果估算 `actuator_delay_s`, `safety_margin_beans`, v1 expected accuracy。

沒有這組數據，任何 dosing architecture 都只能停留在合理猜測。
